# SRS 需求文件審查 Skill

審查並修正 `{batch}-SRS/src/R00XX_*.md` 需求文件，確保符合專案
CLAUDE.md 規範 + 後續 publish 到 Notion 的相容性。本 skill 直接 Edit 修正
檔案，不只是回報。

## 適用範圍

僅處理 `{batch}-SRS/src/R00XX_*.md` 下的獨立需求檔。

**禁止**直接編輯 `{batch}-SRS/output/requirements-{batch}.md` — 該檔由
`srs-sync` skill 負責。

## Batch Resolution（多 batch 推斷）

本 skill 操作對象是「某個 SRS batch」。執行前先決定 **batch ID**：

1. **若使用者明確指定** batch（例如「審查 3-B 的 R0044」、「SRS R0044 in 3-B」），用該 batch
2. **否則**：用 Glob 列出 repo root 下所有 `*-SRS/` 目錄
   - **恰好 1 個** → 自動使用
   - **0 個** → 報錯「找不到任何 SRS batch」
   - **2 個以上** → 報錯，列出可選 batch 並要求明確指定
3. **不要猜**

從 batch ID 推導路徑：
- src 目錄：`{batch}-SRS/src/`
- 目標檔：`{batch}-SRS/src/R00XX_*.md`
- assets：`{batch}-SRS/assets/{R00XX}/`
- output（給 srs-sync 用）：`{batch}-SRS/output/requirements-{batch}.md`

## Workflow

### Step 1: 定位與讀取

1. 用 `Glob {batch}-SRS/src/R00XX_*.md` 找到目標檔案
2. Read 完整檔案內容
3. 若檔案不存在，回報錯誤並停止流程

### Step 1.5: 結構性檢核（腳本）

在進入語意檢核前，先跑結構性 validation 腳本：

```bash
node .claude/scripts/validate-structure.js {batch}-SRS
```

腳本會自動檢查：
- `src/` 下檔名是否符合 `R\d{4}_.+\.md` 規則
- 每份 R 檔是否包含六大章節（一、需求說明 ... 六、待釐清項目）的標題
- `assets/{R00XX}/` 子資料夾是否對應到實際存在的 R-number

**分工原則**：
- **腳本管結構**（檔名、章節是否存在、assets 對應）— 確定性檢查
- **LLM 管語意**（內容歸屬、術語、寫作風格、補寫）— 需要判斷

若腳本回報問題（exit code 1），優先處理腳本指出的結構問題，再進入 Step 2
的語意檢核。腳本檢查的部分**不需要**在 Step 2 重複做。

### Step 2: 十項檢核並修正

依下列清單逐項審查並用 Edit 直接修正檔案。

#### 檢核 1：六大章節結構

確認以下六章節依序完整存在：

1. 一、需求說明
2. 二、使用情境
3. 三、功能需求
4. 四、非功能需求
5. 五、驗收條件
6. 六、待釐清項目（若無，填寫「（無）」）

缺漏章節須補建空殼，內容於檢核 9 補寫。

#### 檢核 2：內容歸屬

**主詞速記：主詞是「人」→ 二、使用情境；主詞是「系統」→ 三、功能需求。**

| 內容性質 | 歸屬章節 | 辨識特徵 |
|----------|----------|----------|
| 功能目的、業務脈絡（解釋「為什麼需要」） | 一、需求說明 | 不描述具體操作 |
| 角色＋動作＋順序 | 二、使用情境 | 主詞是人，常含「→」流程箭頭 |
| 系統行為與規格（系統須提供／支援／自動…） | 三、功能需求 | 主詞是系統，可對應驗收條件 |

混寫者須拆分至各自章節，不可保留在同一條目中。

#### 檢核 3：文字階層格式

| 層級 | 格式 | 用途 |
|------|------|------|
| 第一層 | 一、二、三、 | 大區塊 |
| 第二層 | 1. 2. 3. | 條列功能 |
| 第三層 | i. ii. iii. | 補充條件 |
| 第四層 | - | 細節列表 |

- 同層**不可混用格式**（例如不可在 `1. 2. 3.` 中混入 `(1) (2)`）
- 建議最多展開至三層，避免巢狀過深
- 第二層對應「功能」，第三層對應「規則與條件」

#### 檢核 4：術語合規

逐字檢查全文，將下表「禁止混用」之詞統一為「統一用詞」，使用 Edit 的
`replace_all: true` 批次替換。

| 統一用詞 | 禁止混用 |
|----------|----------|
| 台積電員工（住戶） | 使用者、房客、租戶 |
| 櫃檯人員 | 櫃台人員、前台人員 |
| 駐廠人員 | 現場人員 |
| 承辦人員 | 承辦、管理者 |
| 排房人員 | 配房人員 |
| 清潔人員 | 打掃人員、清潔工 |
| 報修 | 維修申請、修繕申請 |
| 點檢 | 檢查、巡檢 |
| 派工 | 指派、發包 |
| 排房 | 配房、分房 |
| 備品 | 耗材、物資 |

#### 檢核 5：寫作風格

- **主詞明確**：每條需求清楚標示動作主體（如「系統須…」「員工可…」）
- **條件與例外**：逐條列出，禁止以「等」「其他情況類推」帶過
- **可驗證性**：每條功能需求須能對應至「五、驗收條件」中至少一項
- **避免實作細節**：描述「做什麼」而非「如何做」（除非有明確技術限制）
- **用語一致**：同一概念全文使用同一說法

#### 檢核 6：跨需求引用

- 引用格式：`詳見【R00XX】功能名稱`
- 用 Glob 驗證被引用之檔案存在於 `{batch}-SRS/src/`
- 共用流程須指定主要定義處，其餘需求以引用方式帶出
- **注意**：Notion 沒有 markdown anchor 機制，publish 後此格式會變純文字導覽提示（Ctrl+F），非可點擊連結。此為 v1 限制。

#### 檢核 7：圖片引用

- 路徑格式：`../assets/{需求編號}/{圖片檔名}.png`
- 須附說明文字（例如「員工點檢表（一般／中文）」）
- 檔名格式：`{需求編號}_{描述}.png`
- 用 Bash `ls` 確認檔案實際存在
- 不可僅放圖片而無標題說明
- **注意**：Notion MCP 不支援本地圖片自動上傳，publish 時會輸出「需手動上傳」清單

#### 檢核 8：驗收條件

- 編號：`AC-01`、`AC-02` 依序
- 每條須包含三段：**前置條件 → 操作步驟 → 預期結果**
- 須涵蓋正常流程與主要異常流程
- 須能對應至「三、功能需求」中的條目（雙向可追溯）

#### 檢核 9：內容補寫

若章節內容明顯不足或缺漏，依業務脈絡與上下文合理補寫。

**補寫原則**：
- 不可憑空捏造業務規則 — 不確定處須記錄至「六、待釐清項目」
- 補寫之內容須在最終檢核摘要中明確標註，供使用者確認
- 補寫之內容若涉及業務假設，同時新增一條至「六、待釐清項目」

#### 檢核 10：Notion 相容性預檢

為了後續 `srs-publish-notion` skill 能順利 publish，檢查以下語法限制。
發現違規時**直接修正源檔**。

| 項目 | 規則 | 修正方式 |
|------|------|----------|
| Heading 深度 | 最深 H4（Notion H5/H6 自動降為 H4） | 若有 H5/H6，改用 toggle 或降級為 H4 |
| 多行 quote | 多行引用須用 `<br>` 連接 | 將連續 `> ` 行合併為單行 + `<br>` |
| Inline math | `$` 外需空白、內不需 | `$x+y$` → `$x+y$`（前後加空白） |
| Callout 內格式 | 必須用 markdown，不用 HTML | `<strong>` → `**...**` |
| Code block 轉義 | 程式碼內不要轉義 `\[` `\]` | 改回原始字元 |
| 單段字元數 | 單段 rich text < 2000 字元 | 過長段落拆多段 |
| 文章 H1 | 源檔 `# R00XX` H1 **保留**（閱讀用） | 不處理，由 publish skill 剝除 |

### Step 3: 待釐清項目

審查中若發現以下情況，新增至「六、待釐清項目」：

- 業務規則不明確或有多種解讀可能
- 跨需求之流程銜接點不清楚
- 條件分支或例外情境未涵蓋
- 補寫內容中涉及業務假設之處

若無待釐清項目，維持「（無）」。

### Step 4: 輸出檢核摘要

修正完成後，輸出以下格式：

```
## R00XX 檢核摘要

### 修正項目
- ✅ / ⚠️ 六大章節結構
- ✅ / ⚠️ 內容歸屬
- ✅ / ⚠️ 文字階層格式
- ✅ / ⚠️ 術語合規
- ✅ / ⚠️ 寫作風格
- ✅ / ⚠️ 跨需求引用
- ✅ / ⚠️ 圖片引用
- ✅ / ⚠️ 驗收條件
- ✅ / ⚠️ Notion 相容性

### 補寫內容（若有）
- 章節 X：補寫了什麼、補寫原因

### 待釐清項目（若有）
- 項次與內容
```

✅ = 通過無需修改；⚠️ = 有修正並簡述改了什麼。

## 完成檢查

- [ ] 六大章節完整且依序
- [ ] 內容歸屬正確（人 vs 系統）
- [ ] 文字階層與術語符合規範
- [ ] 跨需求引用與圖片路徑可驗證
- [ ] 驗收條件雙向可追溯至功能需求
- [ ] Notion 相容性預檢通過
- [ ] 補寫內容已標註於摘要
- [ ] 待釐清項目已列出或確認為「（無）」
